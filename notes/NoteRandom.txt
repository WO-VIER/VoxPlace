Threads : 

    Contexte Switch (os): Fake multitache (1 coeur physique)
        1. Le thread tourne le cpu lit son rip et fait son F-D-E

        2. L'os envoie un apele system environ toute les mesh

        3. Le cpu s'arrete il prend tout le states de registres et les copies dans la ram dans un structure (Trhead control block)

        4. Il va chercher les registres sauvegardé du thread B dans la ram et les injecte dans les registres

        5. Le program counter pointe maintenent ver le code du thread B. Le fetch reprend

            Un contexte switch coute des miliers de cylcles d' horloges perdu a copier des registres vers la ram
    
    Threads matériels (SMT/ Hyperthreading)
        
            Dans 1 coeur physique ils ont 2 jeux de registres physique et 2 Program Counters

            Un coeur avec 2 threads matériel 

            Le coeur a un seul gros pipeline de fetch-decode-execute. A chaques cycle d'horloge, il regarde les deux Program Counter.
            Il fetch un peut du code du thread 1 et un peut du 2 

            L'éxecution: Si le thread 1 a besoins de l'ALU mais qu'il est occupé par le thread 2, le thread 1 doit attendre.
            Mais si le thread 1 fait une Addition(ALU) et que le thread 2 fait une multiplication flotante (FPU), le cpu envoie les deux opérations aux unités correspondantes en meme temps dans le meme cycle d'horloge
    Un thread != Un process

    en c un thread va attendre un pointeur de fonction mais c'est pas une fonction on va juste éguiller le thread sur un "job"
    
    pthread_create(&thread, NULL, my_meshing_function, chunk_pointer);

        1. Création de sa propre Stack : (1 ou 2 mo)

        2. Le registre RDI (ARGUMENT) : L'OS prend chunk_pointer(argument) et le place physiquement dans le registre RDI
        (Regle x86_64) le premier argument d'une fonction va dans rdi

        3. Le registre RIP (Instruction Pointer) L'os prend le pointeur de fonction (my_meshing_function) et le force dans RIP du cpu

        4. Le cpu regarde son RIP voit l'adresse de la fonction et commence fetch-decode-execute

    Un thread possede son propre Program Conter (RIP) et fait son fetch-decode-execute
